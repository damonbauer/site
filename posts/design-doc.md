date: 2019-12-20

The process of breaking down work has been helpful to me in order to ship features quickly. At a high level, the process looks like this:

1. Define the high level "goal" of the feature
2. Write a user story
3. List out "subtasks" in order to get from point A -> point B

### Define the goal
I like to start with this because it sets the stage for all the work ahead. Usually, I make this a "heading", so it's only a few words. Here are a few examples:

- Build a "Containers" Prototype
- Default Input Validation
- Break Apart UserService Class

With this goal defined, we know what we're aiming for. With that target, you can write a user story.

## User Story
I used to think these were silly and didn't provide any value. Now, I realize that a user story exists so you can continually ground yourself in "why" of your feature and make sure what you're building lines up with what you say is most important.

As an example:

**As a** Marketer

**I want** a simple way to get form validation

**So that** I don't need a developer for our standard forms
